# 开造平台 — 全局业务流程与数据流转

> 版本：2026-04-03 | 用途：团队内部对齐

---

## 一、角色定义

| 角色 | 说明 |
|------|------|
| **项目方** | 有软件需求、发起项目的人（非技术背景为主）、服务 |
| **专家团队方** | 承接需求、提供开发交付的人或团队或openclaw，服务形态不限 |
| **平台** | 开造平台本身，负责撮合、托管、评级 |

---

## 二、系统分工

| 系统 | 职责边界 |
|------|---------|
| **Flutter 前端** (`app/`) | 用户交互、页面渲染、SSE 流式展示 |
| **Go 后端** (`server/`) | 用户体系、项目 CRUD、竞标、支付托管、状态机（撮合/支付/验收） |
| **AI Agent** (`ai-agent/`) | 需求分析、文档生成、智能匹配评分、专家评级、对话助手 |

**关键原则：Go 后端是数据权威方，AI Agent 不创建项目行，只读 `projects` / `users` 表，写 `ai_` 前缀独有表。**

## 三、用户注册与身份建立

### 3.1 登录注册

```
打开 APP → 启动页(Splash) → 引导页(3页产品介绍) → 登录页
                                                      │
                                         ┌────────────┼───────────────┐
                                         ▼            ▼               ▼
                          手机+验证码/ 手机+密码   手机+验证码/ 手机+密码  微信OAuth
                                      登录           注册        			 (预留)
                                         │            │
                                         └─────┬──────┘
                                               ▼
                                         用户不存在？引导注册
                                               │
                                               ▼
                                          选择角色
                                     ┌─────────┴─────────┐
                                     ▼                   ▼
                                  项目方(1)           专家团队方(2)
```

**数据流转**：
- 发送验证码 → Redis 存 `sms:code:{phone}`（5 分钟过期，60 秒发送冷却）
- 登录成功 → `users` 表写入/读取 → 签发 JWT（access_token + refresh_token）
- 选角色 → 更新 `users.role`

### 3.2 需求方引导（4 步）

```
填基本资料(昵称/头像/手机) → 引导创建首个项目(可跳过) → 填写项目信息(标题/描述/分类/预算) → 完成草稿 → 进入首页
```

**数据流转**：更新 `users` 表（nickname, avatar_url, contact_phone） → 可选创建 `projects` 草稿

### 3.3 专家引导（3 步）

```
填资料(昵称/手机/技能/工具/交付效率（简单、中等、复杂）/薪酬范围/接单状态) → 补充自我介绍 → 展示定级结果(Lv.1-4) → 进入首页
```

**数据流转**：
- 更新 `users` 表 + `user_skills` 关联表
- 专家 `onboarding_status` = 1（待审核）
- 两条审核通过路径：
  - **申请审核**：提交简历/作品集 → 等待管理员审批
  - **邀请码**：输入团队邀请码（KZ-xxxx）→ 立即通过，`onboarding_status` = 2

## 四、发布需求（需求方视角）

### 4.1 AI 辅助创建（主路径）

```
首页点"发需求" → 5步发布向导
                    │
  ┌─────────────────┼─────────────────────────────────────┐
  ▼                 ▼                 ▼           ▼       ▼
1.选分类        2.AI对话          3.PRD预览    4.设预算  5.选匹配方式
(dev/visual/   (7-10轮澄清        (自动生成    (滑块+AI  (如何找专家)
 data/solution)  需求)             文档预览)    建议)
                  │
                  │ 每轮澄清：
                  │ 前端发消息 → AI Agent SSE 流式返回
                  │ → text事件(打字效果) → done事件(结构化问题+选项)
                  │ → analysis_complete=true 时显示「确认需求」按钮
                  │
                  ▼
             确认PRD → 发布项目 → 撮合匹配专家团队方成功（招投标选择团队后） → 平台介入对齐需求 →确认合作，触发EARS拆解单元卡片 → 生成requirement.md
```

**数据流转**：
- 对话历史 → Redis（session_id = `req-{project_id}`）
- 维度评分 → AI Agent 内存 → `done` 事件返回前端
- PRD 确认 → `ai_prd_items` 表（需求条目）+ `ai_ears_tasks` 表（原子任务卡）
- 文档 → Minio `projects/{id}/v1/requirement.md` + 本地文件
- 草稿/发布 → Go 后端 `projects` 表（status=1 草稿 / status=2 已发布）

**AI 澄清 7 大维度**：

| 维度 | 权重 | 说明 |
|------|------|------|
| 产品定位 | 20% | 做什么、不做什么 |
| 用户画像 | 15% | 谁用、使用场景 |
| 核心功能 | 20% | 功能模块和关键交互 |
| 技术偏好 | 10% | 平台、技术栈 |
| 商业目标 | 10% | 变现方式 |
| MVP 范围 | 15% | 第一版做哪些 |
| 约束条件 | 10% | 预算、工期 |

所有维度 ≥ 60 且总分 ≥ 80 → 自动触发 PRD 生成

### 4.2 直接创建（简单路径）

```
填标题/描述/分类/预算 → 直接发布
```

**数据流转**：直接写 `projects` 表，无 AI 流程

## 五、AI 文档流水线（需求确认后，文档允许变更后手动上传，重新解析）

```
requirement.md ──确认──→ design.md ──确认──→ task.md
     (PRD+EARS)              (架构设计)          (任务分解)
       │                       │                    │
       │ 每个阶段：             │                    │
       │ start → running →     │                    │
       │ awaiting_confirmation →│                    │
       │ confirmed             │                    │
       ▼                       ▼                    ▼
   前端可查看/下载         前端可查看/下载       task confirmed 后
                                                项目可发布到广场
```

**守卫规则**：requirement ✅ 才能 start design，design ✅ 才能 start task

**数据两层结构**：
```
PRD
├── 功能模块（如：用户系统）
│   ├── 需求条目 F-1.1（需求方看这层，跟踪进度）
│   │   ├── EARS T-001（专家看这层，原子开发任务）
│   │   ├── EARS T-002
│   │   └── EARS T-003
│   └── 需求条目 F-1.2
└── 功能模块（如：课程管理）
```

**数据流转**：
- 阶段状态 → Redis（ProjectState）
- 文档 → Minio + 本地双写
- 文档元数据 → `ai_documents` 表

## 七、撮合与匹配

### 7.1 需求方选人

```
项目详情 → 查看投标列表 → 对比各投标方案
                             │
              ┌──────────────┼──────────────┐
              ▼              ▼              ▼
         手动选择        AI推荐排序       快速匹配
        (从投标中选)    (五维评分排序)   (AI选第一名，直接撮合成功，进入需求确认阶段，需求双方确认后，需要进行确认团队选择)
```

### 7.2 接受投标（核心事务，一次性完成 6 件事）

```
需求方点击「接受」→ 一个事务内同时：
  1. bid.status → 2（已接受）
  2. project.status → 3（进行中），写入 provider_id / agreed_price / matched_at
  3. 创建托管支付订单（orders 表，status=1 待支付）
  4. 创建双方聊天会话 + 发送系统消息
  5. 给需求方发通知（含专家手机号）
  6. 给专家发通知（含需求方手机号）
```

**数据流转**：写 `bids` / `projects` / `orders` / `conversations` / `messages` / `notifications` 六张表

### 7.3 AI 智能匹配评分逻辑

```
读取需求信息(projects) → 生成语义向量 → Milvus召回Top-50专家
     → MySQL补全专家数据(users) → 五维加权评分 → LLM生成推荐理由 → 结果落库
```

| 维度 | 权重 | 数据来源 |
|------|------|---------|
| 技能匹配度 | 30% | 向量相似度 + 技能交集 |
| 历史评价 | 25% | `users.avg_rating` + 完成率 |
| 价格匹配 | 20% | `users.hourly_rate` vs 需求预算 |
| 响应速度 | 15% | `users.response_time_avg` |
| 作品相似度 | 10% | Milvus 向量距离 |

**调节因子**：活跃度时间衰减 × 新人扶持

**数据流转**：推荐结果 → `ai_match_results` 表

## 八、支付与托管（mvp版本不涉及）

```
需求方收到支付提醒 → 订单确认页(金额明细+优惠券) → 选支付方式 → 支付
                                                              │
          ┌───────────────────────────────────────────────────┘
          ▼
  order.status=2(已支付) → 资金进入平台托管
          │
          │ 里程碑验收通过后
          ▼
  释放对应比例资金 → 专家钱包余额增加 → 可提现
```

**费率**：平台抽成 12%，`total_amount = amount × 1.12`

**数据流转**：
- `orders` 表（status: 1待支付 → 2已支付 → 3已释放）
- `wallet` 余额变动 → `transactions` 流水表（1=收入 / 2=提现 / 3=手续费）

## 九、项目执行与管理

### 9.1 PM 方案自动生成

```
撮合成功 → Go后端调用 AI lifecycle hook /on-matched
     → AI Agent 加载3份前序文档 + 商定价格/工期
     → PM Agent 生成 project-plan.md（含里程碑拆分+支付比例）
     → 里程碑数据写入 ai_milestones 表
```

### 9.2 项目管理看板（撮合成功后）

需求方和专家共用管理页，5 个 Tab：

| Tab | 功能 | 谁操作 |
|-----|------|--------|
| 看板 | 三列看板（待办/进行中/已完成），拖拽移动任务 | 专家拖拽，需求方只看 |
| 里程碑 | 时间线展示，含支付比例 | 双方查看 |
| PRD | 需求文档在线阅读 | 双方查看 |
| 文件 | 项目文件管理 | 双方上传/下载 |
| 日报 | 每日进度报告 | 专家填写，需求方查看 |

### 9.3 任务状态流转

```
todo(1) → in_progress(2) → completed(3)
```

### 9.4 里程碑验收

```
专家提交交付物 → 需求方验收页
                     │
              ┌──────┴──────┐
              ▼             ▼
          通过(3)       要求修改(4)
       释放该期资金    附修改意见，专家重做
```

**数据流转**：`milestones` 表（status: 1待开始 → 2进行中 → 3已验收 / 4需修改）

## 十一、评价体系

### 11.1 项目完成后互评

```
项目完成 → 双方可互评
  需求方评专家：质量/沟通/时效/专业度（各项星级）
  专家评需求方：需求清晰度/支付及时/配合度（各项星级）
```

**数据流转**：`reviews` 表，每个项目每人只能评一次

### 11.2 AI 专家评级（vc-T 体系）

```
专家上传简历 → AI解析 → 五维评分 → 计算VibePower积分(满分750) → 定级(vc-T1~T10)
```

**五维权重**（满分 750）：

| 维度 | 权重 | 满分 |
|------|------|------|
| 技术深度 | 30% | 225 |
| 项目经验 | 25% | 187.5 |
| AI 熟练度 | 20% | 150 |
| 作品集 | 15% | 112.5 |
| 教育背景 | 10% | 75 |

**等级体系**：vc-T1(0-99) → vc-T2(100-199) → ... → vc-T10(1900+)，初始化定级上限 vc-T5

**积分变动触发**：

| 行为 | 积分 |
|------|------|
| 完成项目 | +30 ~ +100 |
| 五星好评 | +20 |
| 准时交付 | +15 |
| 提前交付 | +25 |
| 复购客户 | +30 |
| 逾期 | -20 |
| 差评 | -30 |
| 中途退出 | -50 |
| 不活跃衰减 | -5 |
| 投诉成立 | -40 |

**数据流转**：`ai_provider_profiles` 表 + `ai_vibe_power_logs` 流水表

## 十六、SSE 事件协议（前后端对齐）

### 通用事件流

```
init          → {project_id, session_id}
thinking      → "正在理解您的需求..."
tool_call     → "正在生成 PRD 文档..."
tool_result   → "PRD 已生成"
text          → "好的，我来帮你整理一下..."（打字效果）
done          → {question, completeness_score, analysis_complete, duration_ms}
stage_info    → {sub_stage, completeness_score, dimension_coverage}
error         → "错误信息"
```

### done 事件 question 结构

```json
{
  "question": {
    "id": "q3",
    "content": "你最希望优先看到哪些数据？",
    "input_type": "multi_choice",
    "min_select": 1,
    "max_select": 3,
    "options": [
      {"key": "A", "label": "消耗/花费"},
      {"key": "B", "label": "点击量/转化率"},
      {"key": "C", "label": "其他", "is_custom": true}
    ],
    "allow_custom": true,
    "placeholder": "请描述..."
  },
  "completeness_score": 45,
  "analysis_complete": false,
  "duration_ms": 3200
}
```

`input_type` 枚举：`single_choice` / `multi_choice` / `free_text` / `number`

`analysis_complete`：`true` 表示可以输出 PRD，前端可展示"确认需求"按钮
