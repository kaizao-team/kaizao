# 前端交互 EARS 需求卡片审阅与补漏

> 审阅日期：2026-03-23
> 审阅对象：`前端交互EARS需求卡片.md`
> 适用项目：开造 Flutter App（当前仓库 `app/`）

## 1. 结论

这份 EARS 卡片可以作为功能清单用，但还不能直接作为当前项目的前端开发基线。

主要问题有四类：

1. **范围冲突**：MVP/v1/v2 的边界不稳定，且与现有 PRD、UX、代码常量不一致。
2. **文档耦合过深**：行为需求、视觉细节、动效细节、平台实现细节混写在一起。
3. **实现断层**：当前代码还处在壳子阶段，文档却已经写到完整业务态、支付态、AI态、项目协作态。
4. **前置条件缺失**：缺少 API 契约、本地状态、权限处理、失败回退、平台差异说明。

## 2. 当前项目现状映射

当前仓库里，前端已有页面骨架，但离这份卡片定义的交互深度差得很远。

已具备雏形的模块：

- `AUTH`：手机号验证码登录页已存在。
- `ONBOARD`：目前是首启宣传页，不是注册后的角色引导。
- `HOME`：只有需求方首页壳子。
- `POST`：只有 5 步流程壳子，没有真实表单和 AI 会话。
- `CHAT` / `PROF`：只有静态列表和静态个人页。

基本空缺或仅有路由占位的模块：

- `MRKT`、`PRD`、`MTCH`、`PROJ`、`ACCP`、`PAY`、`RATE`、`TEAM`

结论很直接：这份卡片更像“理想态全量交互稿”，不是“当前代码库可直接开工的 Sprint 需求单”。

## 3. 主要问题

### 3.1 MVP 范围和角色模型不一致

冲突点：

- EARS 卡片写明 v1 只保留两个角色，去掉“两者都是”。
- 现有 UX 文档仍保留“需求方 / 供给方 / 两者都是”。
- 现有代码常量和用户模型也仍支持双角色。
- EARS 卡片把 `TEAM`、`RATE` 标成 v2，但首页、撮合和统计汇总里又把相关能力混进 v1 语境。

影响：

- 首页、个人主页、角色切换、引导流程、接口字段都会受影响。
- 不先统一“账号角色”与“当前活跃角色”的定义，后面会反复返工。

建议：

1. 明确单一事实源：`PRD/UX/EARS/代码常量` 只保留一套角色定义。
2. 若 v1 只有两个角色，就删除 `roleDual=3` 的产品定义，或者保留字段但声明前端不暴露。
3. `TEAM`、`RATE` 若延期，就把所有 v1 页面里的相关入口一并降级或隐藏。

### 3.2 文档自身统计和编号不可靠

发现：

- 模块编号里 `M3` 被重复用于 `HOME` 和 `MRKT`。
- `FE-CMN-004` 标题出现异常文本污染。
- 附录统计和正文对不上。

按正文标题自动计数后的结果：

| 模块 | 正文卡片数 | 附录卡片数 | 备注 |
|------|-----------|-----------|------|
| AUTH | 6 | 7 | `AUTH-005` 被划掉但附录仍计入 |
| ONBOARD | 9 | 9 | 类型分布对不上 |
| HOME | 7 | 7 | 类型分布对不上 |
| MRKT | 7 | 7 | 类型分布对不上 |
| PRD | 9 | 9 | 类型分布对不上 |
| MTCH | 5 | 5 | 类型分布对不上 |
| PROJ | 8 | 8 | 类型分布对不上 |
| CMN | 10 | 10 | 类型分布对不上 |

建议：

1. 先修正文档编号、标签和统计，再拿去排期。
2. 附录统计改成脚本自动生成，不要手填。

### 3.3 EARS 不宜写死视觉和动效细节

这轮审阅**不评价视觉方向本身**。你们既然有新的视觉设计，那颜色、渐变、光效、圆角、动效风格都应以新视觉稿为准。

但从需求文档结构上看，当前 EARS 仍然有问题：

- 很多卡片把颜色、像素、边框、动画曲线、持续时间写得过死
- 同一类需求里同时混入“业务行为”和“表现细节”
- 一旦视觉方案更新，需求卡片就会整体过期

建议：

1. EARS 保留**行为验收**和**交互结果**。
2. 颜色、圆角、视觉层级、动效参数改为引用新的 design token / 组件规范。
3. 文档优先级明确为：`产品范围 > 行为需求 > 新视觉规范 > 组件实现`。

### 3.4 引导流程定义和当前代码流程完全不是一回事

当前代码里：

- `Splash` 默认首启走 `onboarding`
- `OnboardingPage` 是 3 页宣传页
- `OnboardingPage._finish()` 还是空实现
- 登录成功后直接进首页，没有角色选择和资料引导

而 EARS 要求的是：

- 首次注册后进入角色选择
- 再进入需求方/专家两条引导链路
- 支持引导断点续传、本地草稿、继续横幅

这不是补几个控件能解决的，是流程基线不同。

建议：

1. 明确区分两种 onboarding：
   - `marketing onboarding`：首启品牌宣传
   - `post-signup onboarding`：注册后角色引导
2. 当前项目建议优先保留后者，前者可弱化甚至去掉。
3. 需要先补一张状态机图：`未登录 / 已登录未选角色 / 已选角色未完成引导 / 已完成引导`。

### 3.5 大量卡片缺少前端可执行前提

以下类型的前置条件没有写清：

- API 返回字段
- 服务端异步状态来源
- 本地缓存键
- 权限申请时机
- iOS / Android 差异
- 失败重试策略
- 降级方案

典型例子：

- `FE-AUTH-003`：短信自动填充在双端都要做，需要原生能力和降级路径。
- `FE-POST-006`：何时显示“生成PRD”，前提是服务端要给出“信息充足”信号。
- `FE-ONBOARD-007`：AI 定级页需要评级接口、轮询/回调、失败态和超时态。
- `FE-PAY-002`：支付唤起、回前台查单、用户取消、订单保活都需要完整支付状态机。
- `FE-CMN-009`：通知权限“在合适时机请求”没有触发条件定义，前端无法实现一致行为。

建议：

每张卡片最少补这 6 个字段：

- `Actor`
- `Entry`
- `API / Event Dependency`
- `Local State`
- `Empty/Error/Fallback`
- `Platform Notes`

## 4. 缺漏项

当前 EARS 卡片缺得最明显的是“系统级约束”。

### 4.1 缺少统一状态机

至少应补 5 组状态机：

1. 登录态
2. 角色态
3. 引导态
4. 发布需求草稿态
5. 支付订单态

### 4.2 缺少 Loading / Empty / Error 的系统覆盖

虽然 `HOME-007`、`CMN-005`、`CMN-003` 提到了少量情况，但仍然不够。

建议为以下模块补统一约束：

- 列表首次加载
- 下拉刷新失败
- 分页失败
- 空列表
- 接口 401/403/404/429/500
- AI 超时/失败
- 上传中断/续传失败

### 4.3 缺少权限与设备能力说明

至少需要明确：

- 短信自动填充
- 相册 / 相机 / 文件读取
- 通知
- 生物认证
- WebView 外链打开
- 支付 App 唤起失败

### 4.4 缺少“当前项目可落地的 MVP 切片”

现有卡片按理想形态写得太满，会直接拖垮前端排期。

建议把 v1 重新切成三层：

- `P0 必须做`
  - AUTH：001/002/004/006/007
  - ONBOARD：001/005/008/009
  - HOME：001/002/003/004
  - POST：001/002/003/006/010
  - CHAT：001/003/006
  - PROF：003/004
  - CMN：002/003/005/006/007/010
- `P1 可上线后补`
  - PRD 双视图
  - MTCH 基础推荐
  - PROJ 简化看板
  - ACCP 基础验收
- `P2 暂缓`
  - TEAM
  - RATE
  - 支付完整闭环
  - 粒子庆祝、复杂手势、双指缩放、全局交互式返回

## 5. 对当前代码的直接影响

### 5.1 应优先修正的代码方向

1. 把注册后角色引导流程做出来，替换当前宣传式 `OnboardingPage`
2. 统一角色模型，只保留 v1 需要的角色和入口
3. 首页拆成需求方 / 专家双视图
4. 把发布需求流程从静态步骤页补成可提交的真实表单流
5. 补公共基础设施：Toast、网络状态、空状态、草稿缓存、权限封装

### 5.2 现在不该急着做的东西

- 支付动效
- 生物认证
- 双指缩放 PRD 树图
- 看板拖拽
- 复杂庆祝动画
- 组队大厅
- 双向评价

## 6. 推荐的修订动作

建议按这个顺序处理原卡片：

1. 先统一角色、MVP、v2 边界
2. 把写死的视觉要求改成引用新的视觉 token 和组件规范
3. 给每张卡片补依赖字段和失败回退
4. 把附录统计改成自动生成
5. 重新切一次真正可开发的 v1 清单

## 7. 一句话判断

这份卡片方向对，但现在还是“全量愿景版”。如果直接拿来驱动当前 Flutter 项目开发，前端会在第一个迭代里同时踩中范围漂移、实现断层和接口不齐三件事。
