## 1. 执行摘要

### 一句话定位

**开造 VCC 是中国首个 Vibe Coding 撮合平台，用 AI Agent 连接"有想法的人"与"会用 AI 编程的人"，让软件开发的成本降低 80%、周期缩短 90%。**

### 核心问题

中国每年有数百万中小企业和个人创业者需要软件开发服务，但传统外包价格高（万元起步）、周期长（数月）、沟通成本极高。与此同时，Vibe Coding（AI 辅助编程）正在催生大量能用自然语言生成代码的新型开发者，但他们缺乏系统化的获客和变现渠道。

**供需两端存在巨大的信息不对称和信任缺失。**

### 解决方案

开造 VCC 通过三大 AI Agent（需求分析 Agent、项目管理 Agent、质量检查 Agent）贯穿全流程：
- **需求方**：用大白话描述需求 → AI 自动生成结构化 PRD 和 EARS 任务卡片 → 智能匹配开发者 → 担保交易 → 分阶段验收
- **供给方**：获得清晰的需求文档 → 高效开发 → AI 辅助质量检查 → 安全收款

### 商业模式

平台佣金（10-15%）+ 增值服务（急速匹配、置顶推广）+ 会员订阅 + 技能认证

### 市场规模

- **TAM**（全球 AI 编程助手市场）：$85 亿 / 约 ¥600 亿（2026 年）
- **SAM**（中国灵活用工 + 软件外包市场中的 AI 编程渗透部分）：¥120 亿
- **SOM**（3 年内可触达市场）：¥3-5 亿

### 融资需求

- **融资轮次**：天使轮 / Pre-A 轮
- **融资金额**：¥500-800 万人民币
- **资金用途**：产品研发（45%）、市场推广（25%）、团队建设（20%）、运营储备（10%）
- **预期使用周期**：12-15 个月，达成 10 万注册用户、月 GMV ¥300 万

## 3. 产品与技术

### 3.1 产品核心功能

开造 VCC 是一款同时支持 Android 和 iOS 的撮合平台 APP，核心功能围绕"AI Agent 驱动的全流程交付保障"展开：

| 核心模块 | 功能亮点 | 差异化价值 |
|---------|---------|----------|
| **AI 需求拆解引擎** | 用户用自然语言描述需求 → AI 多轮对话澄清 → 自动生成结构化 PRD + EARS 任务卡片 + 工期/成本预估 | 解决"需求不清晰"这一行业第一痛点 |
| **智能撮合系统** | 5 维加权匹配算法（技能 30% + 评价 25% + 价格 20% + 响应 15% + 作品 10%），支持主动投标、智能推荐、一键派单 | 撮合效率远超人工浏览 |
| **项目管理看板** | 双视图展示（概览视图 + EARS 卡片视图）、AI 进度提醒、每日简报、延期风险预警 | 交付过程全透明，降低沟通成本 |
| **担保交易系统** | 预付款 → 担保 → 分阶段验收 → 释放，接入微信支付/支付宝电商收付通 | 资金安全有保障 |
| **AI 质量检查** | 功能完整性检查、代码规范检查、响应式适配检查、性能基线检查 | AI 降低验收争议率 |
| **供给方组队系统** | 角色标签体系、智能队友推荐、团队投标、收入自动分账 | 解决个人能力短板，承接复杂项目 |

### 3.2 AI 技术壁垒分析

开造的核心技术壁垒在于**三大 AI Agent 的协同工作**：

| AI Agent | 核心技术 | 壁垒来源 |
|---------|---------|---------|
| **需求分析 Agent** | LLM（Claude/通义千问） + RAG（历史项目知识库） + EARS 结构化输出 + 多轮对话引导 | 随平台交易数据积累，RAG 知识库形成**数据飞轮**——项目越多，需求拆解越精准，吸引更多用户 |
| **项目管理 Agent** | 事件驱动架构 + 延期风险预测模型 + 自然语言日报生成 | 基于平台独有的项目进度数据训练风险模型，**数据壁垒**随时间增强 |
| **质量检查 Agent** | 自动化测试管线（Playwright + Lighthouse + Semgrep） + LLM 对照 EARS 卡片验收 | 检查标准与 EARS 体系深度绑定，形成**方法论壁垒** |

**核心技术优势**：
1. **EARS 需求卡片体系**：将模糊需求转化为原子级、无歧义、可验收的任务单元，是行业内首创的 AI 需求工程方法论在撮合平台的应用
2. **RAG 知识库飞轮**：每一笔完成的交易都会沉淀为知识库数据，持续提升需求拆解和工时预估的准确性
3. **混合模型策略**：复杂任务用 Claude（长上下文、强推理），简单任务用通义千问/DeepSeek（低成本、合规），实现性能与成本的最优平衡

### 3.3 技术架构优势

| 架构决策 | 选型 | 优势 |
|---------|------|------|
| 跨平台 | Flutter 3.x + Dart | 一套代码双端，8 周 MVP，看板拖拽等复杂 UI 原生支持好 |
| 后端 | Go 微服务 + gRPC | 高并发（goroutine 天然适配 IM 系统）、低资源占用、部署简单 |
| AI 服务 | Python + LangGraph + LlamaIndex | LLM/RAG 工具链生态最成熟，通过 gRPC 与 Go 服务通信 |
| 数据库 | PostgreSQL + Redis + Milvus + ES | ACID 保障交易安全、JSONB 存储 EARS 卡片、向量检索支持 RAG |
| 基础设施 | 阿里云 + Docker + K8s | 国内云服务市占率第一，Go/Flutter 生态支持好 |

**MVP 阶段月度基础设施成本约 ¥4,750/月**，含计算、数据库、缓存、搜索、存储、CDN、短信、AI 推理等全部服务。

### 4.3 单位经济模型

| 指标 | 数值 | 计算逻辑 |
|------|------|---------|
| **平均客单价（AOV）** | ¥1,500 | 基于需求方预算区间加权平均：(¥500 x 40% + ¥1,500 x 35% + ¥5,000 x 20% + ¥10,000 x 5%) |

| **用户生命周期** | 2 年 | 保守估计 |
| **LTV（用户生命周期价值）** | ¥1,188 | ¥180 x 3.3 x 2 |
| **CAC（获客成本）** | ¥50 | 初期以内容营销为主（小红书/即刻/社区），CAC 较低 |

| **CAC 回本周期** | **< 1 个月** | 首单即可覆盖 CAC（¥180 > ¥50） |

> 注：以上为理想状态下的单位经济模型。实际运营中，初期补贴（首单五折、免佣等）会降低实际佣金收入，CAC 回本周期可能延长至 2-3 个月。

### 5.2 差异化优势

**开造 vs 传统外包平台（猪八戒/程序员客栈）**：
1. **AI 原生**：AI Agent 贯穿全流程，不是在传统模式上"加 AI"，而是从 Day 1 就围绕 AI 设计
2. **客单价更低、周期更短**：Vibe Coding 使开发成本降至传统外包的 1/5，周期缩短至 1/10
3. **需求标准化**：EARS 卡片体系消除需求歧义，降低返工率和争议率
4. **新供给群体**：服务的是新一代 Vibe Coder，而非传统程序员

**开造 vs 国际平台（Upwork/Fiverr）**：
1. **本土化**：支持微信支付/支付宝、中文 AI Agent、国内合规
2. **垂直深度**：100% 聚焦 Vibe Coding，而非大而全的自由职业平台
3. **全流程 AI**：不只是撮合，而是从需求拆解到质量检查的全链条 AI 赋能

**开造 vs Lovable**：
1. **互补关系而非竞争**：Lovable 是工具层（AI 生成代码），开造是服务层（连接人和需求）
2. **Lovable 用户正是开造的供给方**：会用 Lovable 的人可以在开造上接单变现
3. **开造解决 Lovable 不能解决的问题**：复杂项目需要人类参与、需求方不会用工具、需要交付保障

### 5.3 竞争壁垒

| 壁垒类型 | 具体内容 | 构建时间 |
|---------|---------|---------|
| **数据壁垒** | 交易数据 → RAG 知识库 → 更精准的需求拆解和匹配 → 更多交易 → 更多数据（飞轮效应） | 6-12 个月 |

### 7.3 敏感性分析

**核心变量对年利润（Y2）的影响**：

| 变量 | 基准值 | 乐观（+20%） | 悲观（-20%） | 利润变动幅度 |
|------|--------|-------------|-------------|------------|
| **月成交订单数** | 5,000 | 6,000 → 利润 ¥1,033 万 | 4,000 → 利润 ¥457 万 | ±38.7% |
| **平均客单价** | ¥2,000 | ¥2,400 → 利润 ¥1,033 万 | ¥1,600 → 利润 ¥457 万 | ±38.7% |
| **佣金率** | 12% | 14.4% → 利润 ¥985 万 | 9.6% → 利润 ¥505 万 | ±32.2% |
| **人力成本** | ¥600 万 | ¥720 万 → 利润 ¥625 万 | ¥480 万 → 利润 ¥865 万 | ±16.1% |
| **获客效率（注册→成交转化率）** | 1.5% | 1.8% → 同"订单数 +20%" | 1.2% → 同"订单数 -20%" | ±38.7% |

**关键结论**：
- 收入最敏感于**订单量和客单价**，需优先保障撮合成功率和交易转化
- 人力成本影响相对可控，可通过控制招聘节奏调节
- 即使在悲观情景下（所有变量 -20%），Y2 仍可维持正向利润

**极端压力测试**：
- 若 Y1 实际 GMV 仅达预期的 50%（¥750 万），总收入约 ¥115 万，亏损约 ¥247 万
- 以 ¥800 万融资计算，现金储备可支撑至 M15-M18，有充足时间调整策略

| 4 | **交付质量争议** — 供给方交付物不符预期，产生纠纷 | 高 | 中 | EARS 卡片明确验收标准；AI 质量检查 Agent 自动预检；分阶段验收降低风险；建立仲裁机制 |
| 5 | **供给方绕过平台交易** — 双方私下交易，平台损失佣金 | 中 | 中 | IM 系统敏感词检测；建立平台不可替代价值（AI 工具 + 信用背书 + 担保交易）；维持合理佣金率 |
| 6 | **大厂入场竞争** — 字节/阿里/腾讯等推出类似产品 | 中 | 高 | 速度取胜，抢先占据品类心智；深耕垂直场景建立数据和社区壁垒；大厂做大平台不会聚焦垂直赛道 |
| 7 | **Vibe Coding 热度下降** — 概念泡沫破裂，市场萎缩 | 低 | 高 | 产品定位可扩展为"AI 辅助开发服务平台"，不局限于 Vibe Coding 标签；核心价值是撮合效率而非单一技术标签 |
| 8 | **AI 服务成本过高** — LLM API 调用成本侵蚀利润 | 中 | 中 | 混合模型策略（简单任务用国产模型，成本仅为 Claude 的 1/10）；缓存常见模式；设置单用户日调用限额 |
| 9 | **支付合规风险** — 担保交易涉及资金沉淀，触及监管红线 | 中 | 高 | 全程使用微信支付/支付宝「电商收付通」，资金不进平台账户；不碰资金沉淀；法务顾问前置审核 |
| 10 | **知识产权纠纷** — AI 生成代码的版权归属不清 | 低 | 中 | 服务协议明确 IP 归属条款（交付物归需求方所有）；AI 生成内容版权声明；必要时引入法律顾问 |

### 10.2 法律合规要点

| 合规领域 | 要求 | 应对方案 |
|---------|------|---------|
| **ICP 备案** | 互联网信息服务需 ICP 备案 | 注册公司后立即申请，预计 2-4 周 |
| **APP 上架合规** | 隐私政策、用户协议、未成年人保护 | 参照《APP 违法违规收集使用个人信息行为认定方法》逐项合规 |
| **个人信息保护法（PIPL）** | 最小化收集、明示同意、数据删除权 | 首次启动隐私弹窗；仅收集必需信息；用户注销 30 天内删除数据 |
| **数据安全法** | 数据分级分类保护 | 敏感字段 AES-256 加密；密钥使用阿里云 KMS 托管 |
| **支付合规** | 不得无证从事支付业务 | 接入持牌第三方支付（微信/支付宝），不碰资金沉淀 |
| **税务合规** | 供给方收入个税代扣代缴问题 | 初期以信息服务费模式运营；规模化后考虑灵活用工税务方案 |
| **劳动关系** | 平台与供给方的劳动关系认定风险 | 明确"信息撮合"定位，供给方为独立服务提供者，非雇佣关系 |
