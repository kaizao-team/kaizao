## 1. 产品概述与愿景

### 1.1 产品定位

开造 是一款面向中国市场的 **Vibe Coding 撮合平台** APP，同时支持 Android 和 iOS。平台连接两类核心用户：

- **需求方**：有软件开发需求但预算有限的个人、创业者和中小企业
- **供给方**：熟练掌握 Vibe Coding 工具链的个人开发者或小型团队（"一人公司"）

平台以 **AI Agent** 为核心驱动力，贯穿需求拆解、项目管理、质量检查全流程，确保"模糊想法"能高效转化为"可交付产品"。

### 1.2 核心价值主张

| 角色 | 核心价值 |
|------|---------|
| 需求方 | 用传统开发 1/5 的成本、1/10 的时间，获得可用的软件产品 |
| 供给方 | 低门槛接单变现，AI 辅助提升交付效率，构建个人品牌 |
| 平台 | AI Agent 保障交付质量，降低沟通成本，提升撮合成功率 |

### 1.3 愿景声明

**让每一个好想法都能被低成本实现。**

我们相信 Vibe Coding 正在重新定义软件开发的成本结构。开造 要成为中国最大的 Vibe Coding 服务撮合平台，让"一人公司"成为主流开发力量，让每个有想法的人都能以极低成本将创意变为现实。

### 1.4 市场背景

- 全球 AI 编程助手市场规模：**$85亿**（2026年）
- Vibe Coding 搜索量增长：**6,700%**
- 中国市场现状：**完全空白**——无对标 Upwork/Fiverr vibe coding 分类的本土平台
- 一人公司趋势：Anthropic CEO 预测2026年1-2人可运营十亿美元级公司

## 3. 核心功能模块

### 3.1 需求发布系统

**目标**：让需求方用最简单的方式描述需求，AI 辅助补全和结构化。

| 功能点 | 描述 |
|--------|------|
| 结构化需求模板 | 按类别提供模板：APP开发、网页、小程序、设计、数据分析、技术指导等 |
| AI 辅助填写 | 用户输入简单描述，AI 自动补充技术需求、验收标准、参考案例 |
| 需求预估 | AI 根据需求复杂度自动预估价格区间和工期 |
| 参考案例库 | 展示类似项目案例，帮助需求方明确期望 |
| 附件上传 | 支持图片、设计稿、文档等辅助说明 |

### 3.2 AI 需求拆解引擎（核心差异化）

**目标**：将模糊需求转化为结构化 PRD + EARS 级别的可执行任务卡片，同时提供整体概览视图。

**工作流程**：
```
用户模糊描述 → AI 多轮对话澄清 → 结构化 PRD → 功能模块拆解 → EARS 任务卡片 → 工期/成本预估
```

| 能力 | 描述 |
|------|------|
| 智能问答 | 通过多轮对话引导用户澄清需求盲区 |
| PRD 自动生成 | 输出包含功能列表、用户故事、验收标准的结构化文档 |
| **EARS 任务卡片** | 将 PRD 拆解为符合 EARS 规范的原子级需求卡片（详见下方） |
| 技术栈建议 | 根据需求推荐最适合 vibe coding 的技术方案 |
| 复杂度评估 | 自动评估项目复杂度（S/M/L/XL），关联价格区间 |

#### 3.2.1 EARS 需求卡片体系

**EARS（Easy Approach to Requirements Syntax）** 将每条需求写成结构化的、无歧义的语句，分为五种模式：

| EARS 模式 | 句式模板 | 适用场景 | 示例 |
|-----------|---------|---------|------|
| **Ubiquitous（始终）** | 系统应当 [行为] | 系统持续生效的能力 | "系统应当对所有API请求进行身份验证" |
| **Event-driven（事件）** | 当 [触发事件] 时，系统应当 [行为] | 某个事件触发的响应 | "当用户提交需求时，系统应当在30秒内生成结构化PRD" |
| **State-driven（状态）** | 当 [系统处于某状态] 时，系统应当 [行为] | 系统处于特定状态下的表现 | "当项目处于逾期状态时，系统应当每日推送风险提醒" |
| **Optional（可选）** | 当 [功能被启用] 时，系统应当 [行为] | 可配置的功能 | "当用户启用AI辅助时，系统应当自动补全验收标准" |
| **Unwanted（异常）** | 如果 [异常条件]，系统应当 [处置] | 容错和异常处理 | "如果支付超时，系统应当自动冻结订单并通知双方" |

**每张 EARS 卡片包含**：

```
┌─────────────────────────────────────┐
│ #T-042  [Event] 前端 · 预计 60min   │
│─────────────────────────────────────│
│ 需求：当用户上传设计稿时，系统应当    │
│ 自动识别页面元素并生成组件清单        │
│─────────────────────────────────────│
│ 验收标准：                           │
│ ✓ 支持 Figma/Sketch/PNG 格式        │
│ ✓ 识别准确率 > 80%                   │
│ ✓ 生成结果可人工修正                  │
│─────────────────────────────────────│
│ 依赖：#T-040 · 阻塞：#T-045         │
│ 所属模块：需求发布系统                │
└─────────────────────────────────────┘
```

#### 3.2.2 双视图展示设计（降低认知压力）

**核心设计原则**：信息渐进呈现，避免一次性展示全部细节造成压迫感。

| 视图 | 面向角色 | 展示内容 | 设计要点 |
|------|---------|---------|---------|
| **概览视图（默认）** | 需求方+供给方 | 项目整体结构图：模块→子功能的树状/脑图视图，每个节点仅显示名称+状态色块+预估工时 | 像看目录一样轻松，一屏掌握全貌。绿色=完成，蓝色=进行中，灰色=待开始 |
| **卡片视图（按需展开）** | 供给方为主 | 点击模块节点展开该模块下的 EARS 卡片列表，支持按状态/角色/优先级筛选 | 只看与自己相关的卡片，不被其他模块干扰 |

**降低压力的具体策略**：

1. **渐进式披露** — 默认只展示模块级概览（5-8个模块卡片），不直接展示全部 EARS 卡片
2. **进度驱动视觉** — 用进度环/进度条替代数字堆砌，让"完成了多少"比"还剩多少"更突出
3. **角色过滤** — 供给方只看分配给自己的卡片（团队模式下只看自己负责的角色卡片）
4. **智能折叠** — 已完成的卡片自动折叠到"已完成"区域，保持当前视野清爽
5. **里程碑聚焦** — 当前里程碑高亮，未来里程碑置灰淡化，引导用户专注当下
6. **AI 摘要** — 每个模块提供一句话AI摘要："本模块共8个任务，已完成5个，预计明天完工"
7. **需求方简化视图** — 需求方看到的是"功能清单+完成状态"，不暴露EARS技术细节

### 3.3 智能撮合系统

**目标**：基于多维度匹配算法，为需求找到最合适的供给方。

| 匹配维度 | 权重 | 说明 |
|----------|------|------|
| 技能匹配度 | 30% | 供给方技能标签 vs 需求技术要求 |
| 历史评价 | 25% | 综合评分、按时交付率、好评率 |
| 价格匹配 | 20% | 供给方报价 vs 需求方预算 |
| 响应速度 | 15% | 平均响应时间、在线状态 |
| 作品相似度 | 10% | 历史作品与当前需求的相似程度 |

**撮合模式**：
- **主动投标**：供给方浏览需求列表，主动报价投标
- **智能推荐**：系统主动推送匹配的需求/供给方
- **一键派单**：需求方选择"急速匹配"，系统自动分配最佳供给方

### 3.10 供给方组队系统

**目标**：让擅长不同领域的 vibe coder 自由组队，互补能力短板，共同承接复杂项目。

**设计背景**：
Vibe coding 实际使用者很多是后端开发者，他们在产品设计、UI/UX方面存在短板。同样，设计师可能缺乏后端能力。组队功能让不同角色的供给方形成临时或固定团队，为需求方提供更完整的交付能力。

| 功能点 | 描述 |
|--------|------|
| **角色标签体系** | 供给方标注自己的核心角色：后端开发、前端开发、UI设计、产品设计、数据分析、技术指导等 |
| **组队大厅** | 供给方可发布"寻找队友"帖，说明项目类型、需要什么角色、预期分成比例 |
| **智能队友推荐** | AI 根据项目需求自动推荐互补角色的供给方（如后端 vibe coder 自动匹配 UI 设计师） |
| **团队主页** | 组建后的团队拥有独立主页，展示团队成员、综合技能覆盖、团队作品集 |
| **团队投标** | 团队可以整体投标项目，需求方一次对接一个团队而非多个个人 |
| **内部任务分配** | 团队队长可在项目看板中按成员分配任务，各成员独立推进 |
| **收入自动分账** | 项目验收后按预设比例自动将款项分配给各团队成员 |
| **临时组队 vs 固定团队** | 支持两种模式——按项目临时组队（项目结束自动解散）或创建固定团队（长期协作） |
| **团队评价** | 团队整体评价 + 成员个人贡献评价，避免搭便车 |
| **AI 组队建议** | 需求分析Agent拆解需求后，自动识别项目所需角色，主动建议供给方组队 |

**组队流程**：
```
供给方浏览项目 → AI提示"该项目需要UI设计能力，建议组队"
  ↓
进入组队大厅 → 查看匹配的队友推荐 / 发布寻人帖
  ↓
邀请队友 → 确认分工和分成比例 → 组建团队
  ↓
以团队身份投标 → 中标后在看板内协作 → 分阶段交付
  ↓
验收通过 → 收入按比例自动分账
```

## 4. 用户旅程

### 4.1 需求方旅程

```
注册/登录
  ↓
发布需求（选择类别→填写描述→AI辅助补全→预览PRD）
  ↓
AI 自动生成结构化 PRD + 任务列表 + 价格/工期预估
  ↓
确认需求文档 → 选择撮合模式（主动选择/智能匹配/急速派单）
  ↓
收到供给方投标/匹配推荐 → 查看主页、作品、评价 → 选定供给方
  ↓
预付款到担保账户 → 项目启动
  ↓
在项目看板追踪进度 → IM沟通 → 查看每日简报
  ↓
分阶段验收（预览→测试→确认→付款释放）
  ↓
最终验收 → 双向评价 → 完成
```

### 4.2 供给方旅程

```
注册/登录 → 完善个人信息 → 技能认证
  ↓
浏览需求列表 / 收到智能推荐通知
  ↓
查看 AI 生成的 PRD 和任务列表 → 评估可行性
  ↓
（可选）AI提示需要组队 → 进入组队大厅 → 邀请互补队友 → 确认分工和分成
  ↓
个人或团队投标报价（价格+工期+方案简述）
  ↓
被选中 → 确认项目 → 项目启动
  ↓
在项目看板领取任务（团队模式下按成员分配）→ 开发 → 提交阶段性成果
  ↓
需求方验收 → 修改（如有）→ 通过 → 收到阶段付款
  ↓
最终交付 → 收到尾款（团队模式自动分账）→ 双向评价 → 作品入库
```

## 5. AI Agent 能力定义

### 5.1 需求分析 Agent

| 维度 | 描述 |
|------|------|
| **输入** | 用户自然语言描述（文字/语音），参考图片/链接 |
| **输出** | 结构化 PRD（功能列表、用户故事、验收标准、技术建议）、任务列表、复杂度评估、价格预估 |
| **核心能力** | 多轮对话引导、需求盲区识别、自动补全验收标准、同类项目对标、技术可行性初评 |
| **能力边界** | 不替代专业的系统设计，复杂业务逻辑需人工介入；不做最终定价，只提供参考区间 |
| **技术方案** | LLM（Claude/GPT-4）+ RAG（历史项目知识库）+ 结构化输出 |

### 5.2 项目管理 Agent

| 维度 | 描述 |
|------|------|
| **自动化能力** | 任务状态自动追踪、延期预警、每日简报生成、阻塞点识别 |
| **提醒机制** | 里程碑到期提醒（提前24h）、任务超时提醒、长时间无进展预警 |
| **风险预警** | 基于历史数据预测延期风险，自动建议调整方案 |
| **沟通辅助** | 自动生成进度更新文案、翻译技术术语为通俗表达 |

### 5.3 质量检查 Agent

| 维度 | 描述 |
|------|------|
| **检查维度** | 功能完整性（vs PRD）、代码基本规范、响应式适配、基础安全检查 |
| **评分标准** | 功能覆盖率（权重40%）、代码质量（20%）、UI还原度（20%）、性能基线（20%） |
| **自动检查** | 链接可用性、页面加载速度、基本SEO、移动端适配 |
| **能力边界** | 不替代专业QA，复杂业务逻辑测试仍需人工；安全检查仅覆盖OWASP基础项 |

## 7. 技术架构建议

### 7.1 跨平台方案

| 方案 | 推荐度 | 理由 |
|------|--------|------|
| **Flutter** | ★★★★★ 推荐 | 一套代码双端、性能接近原生、UI一致性好、Dart生态成熟、适合快速迭代 |
| React Native | ★★★☆☆ | 生态大但性能不如Flutter、桥接开销、UI一致性需额外工作 |
| 原生开发 | ★★☆☆☆ | 双端维护成本过高，不适合初创团队 |

**推荐：Flutter + Dart**

### 7.2 后端架构

```
├── API Gateway（Nginx/Kong）
├── 核心服务（Go/Node.js 微服务）
│   ├── 用户服务（注册/登录/认证）
│   ├── 需求服务（发布/管理/搜索）
│   ├── 撮合服务（匹配算法/推荐）
│   ├── 项目服务（看板/进度/里程碑）
│   ├── 交易服务（支付/结算/担保）
│   ├── 消息服务（IM/通知/推送）
│   └── 评价服务（评分/信用/等级）
├── AI Agent 服务（Python）
│   ├── 需求分析引擎
│   ├── 项目管理引擎
│   └── 质量检查引擎
├── 数据层
│   ├── PostgreSQL（核心业务数据）
│   ├── Redis（缓存/会话/实时状态）
│   ├── Elasticsearch（搜索/推荐）
│   └── 对象存储（OSS，文件/图片）
└── 基础设施
    ├── 阿里云/腾讯云
    ├── Docker + K8s
    └── CI/CD Pipeline
```

### 7.3 AI Agent 技术栈

| 组件 | 技术选型 |
|------|---------|
| LLM | Claude API / 通义千问（国内合规备选） |
| RAG | 向量数据库（Qdrant/Milvus）+ 历史项目知识库 |
| Prompt 工程 | 结构化输出、Chain-of-Thought、Few-shot |
| Agent 框架 | LangChain / 自研轻量框架 |
| 输出格式 | Markdown PRD + JSON 任务列表 |

## 10. 风险与应对

| # | 风险 | 影响 | 概率 | 应对策略 |
|---|------|------|------|---------|
| 1 | **供给方冷启动不足** | 需求方找不到人接单 | 高 | 从小红书/即刻社区定向招募种子用户；初期平台补贴，降低佣金至5% |
| 2 | **需求方不付款/跑路** | 供给方利益受损 | 中 | 担保交易强制预付；分阶段付款降低单次风险 |
| 3 | **AI生成的PRD质量不够** | 用户失去信任 | 中 | 持续优化Prompt+RAG；提供人工审核选项；收集反馈闭环 |
| 4 | **交付质量争议** | 平台口碑受损 | 高 | 明确验收标准、AI辅助质量检查、专业仲裁团队 |
| 5 | **供给方绕过平台交易** | 平台损失佣金 | 中 | 沟通系统内敏感词检测；建立平台价值（AI工具、信用背书）使用户不愿离开 |
| 6 | **支付牌照/合规风险** | 运营受阻 | 中 | 初期接入持牌第三方支付（微信支付/支付宝）；不碰资金沉淀 |
| 7 | **Vibe coding热度下降** | 市场萎缩 | 低 | 定位扩展为"AI辅助开发平台"，不局限于vibe coding标签 |
| 8 | **大厂入场竞争** | 市场份额被抢 | 中 | 深耕垂直场景，建立社区壁垒和数据壁垒；速度取胜，先建立品牌心智 |
| 9 | **AI服务成本过高** | 影响盈利 | 中 | 采用混合模型策略（简单任务用轻量模型，复杂任务用高级模型）；缓存常见模式 |
| 10 | **知识产权纠纷** | 法律风险 | 低 | 明确服务协议中的IP归属条款；AI生成内容的版权声明 |
